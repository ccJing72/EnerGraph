# EnerGraph 项目介绍文档

## 1. 一句话介绍

**EnerGraph** 是面向福加能碳管理平台的能源管理 AI Agent，基于 **LangGraph** 构建，定位为青山大模型 V3.0 五层架构中的**第 3 层决策层**，负责把用户自然语言问题转化为可执行的工具调用、数据查询、知识检索、页面跳转和解释报告。

它不是一个单纯聊天机器人，而是一个连接“用户意图、业务知识、平台数据、算法模型、前端页面”的智能编排中枢。

## 2. 项目背景与产品定位

福加能碳管理平台已经具备大量能源、碳排、光伏、报警、设备、环境等业务数据和页面能力，但传统交互方式依赖用户理解菜单、指标和查询路径。EnerGraph 的目标是把这些能力包装成自然语言入口，让用户可以直接提出问题，例如：

- “查看当前站点 COP”
- “有哪些活跃报警”
- “光伏本月发电情况怎么样”
- “冷冻水系统有什么节能建议”
- “跳转到能耗监测页面”

项目在产品层面承担三个角色：

1. **业务问答助手**：基于 HVAC 知识库回答暖通空调、节能、故障诊断类问题。
2. **运营数据助手**：调用福加平台真实后端接口，查询 COP、能耗、报警、碳排、光伏、设备排名等数据。
3. **前端导航助手**：结合用户意图和业务数据查询结果，向前端下发 `UIAction` 页面跳转信号。

从系统定位看，EnerGraph 位于 LLM 和业务系统之间，重点不是直接做复杂数值优化，而是进行**意图理解、任务拆解、工具调度、结果整合、决策解释**。

## 3. V3.0 五层架构中的位置

根据项目文档，青山大模型 V3.0 可抽象为五层：

| 层级 | 职责 | EnerGraph 关系 |
| --- | --- | --- |
| 第 1 层：用户交互层 | 前端页面、语音、可视化入口 | EnerGraph 通过 FastAPI SSE 与前端对接 |
| 第 2 层：大模型基础层 | LLM 推理、Function Calling、自然语言生成 | EnerGraph 调用 OpenAI / DeepSeek / Anthropic 兼容模型 |
| **第 3 层：决策层** | 意图理解、工具调度、业务编排、解释生成 | **EnerGraph 的核心定位** |
| 第 4 层：算法模型层 | 负荷预测、光伏预测、储能调度、物理校验等 | 当前通过 MCP 接口规范预留，尚未真实接入 |
| 第 5 层：数据与设备层 | 福加平台、设备数据、能耗数据、告警、站点信息 | 当前已通过 Java 后端 API 工具接入部分真实数据 |

因此，EnerGraph 的核心价值是：**把大模型的语言理解能力与能源管理平台的真实业务能力连接起来，并用可解释、可追踪、可扩展的方式执行。**

## 5. 总体架构设计

项目采用分层架构，主要分为配置层、Graph 调度层、Skills 业务推理层、Tools 原子执行层、Schemas 数据模型层、API 服务层和前端演示层。

```text
用户 / 前端页面
    │
    │ HTTP / SSE
    ▼
FastAPI 服务层
    │
    ▼
LangGraph 决策图
    │
    ├── cognitive_parser：理解用户意图，选择工具
    ├── v3_engine_router：执行工具，调度 Skill
    └── interpreter_generator：整合数据，生成报告
    │
    ▼
Skills 业务推理层
    │
    ├── HVACExpertSkill：暖通知识问答
    ├── UIRouterSkill：页面跳转推断
    ├── EnergyDispatchSkill：能源调度骨架
    └── V3InterpreterSkill：解释器骨架
    │
    ▼
Tools 原子执行层
    │
    ├── HVAC RAG 检索
    ├── 福加 Java 后端 API
    ├── 页面跳转工具
    └── 业务意图解析工具
    │
    ▼
知识库 / 福加平台 / 未来 MCP 算法模型
```

该设计的核心思想是：

- **Graph 只负责流程编排**，不沉淀具体业务逻辑。
- **Skill 负责业务 SOP**，例如 HVAC 低置信拒答、页面跳转推断。
- **Tool 负责确定性执行**，例如查知识库、查接口、构造跳转动作。
- **Schema 负责输入输出契约**，保证 API、Graph、Tool 之间的数据结构稳定。
- **Prompt 配置外置**，降低提示词和代码逻辑的耦合。

## 6. 核心运行流程

主流程由 `src/graph/builder.py` 组装，入口为 `cognitive_parser`。

```text
开始
  │
  ▼
cognitive_parser
  │
  ├── 如果 LLM 需要调用工具 → v3_engine_router
  │
  └── 如果 LLM 直接回答 → interpreter_generator
  │
  ▼
v3_engine_router
  │
  ├── 执行 TOOL_REGISTRY 中的工具
  ├── 将工具结果写入 AgentState
  ├── 根据工具名匹配 BaseSkill
  └── 执行 Skill 的 before_execute / execute / after_execute
  │
  ▼
cognitive_parser
  │
  └── 继续判断是否还需要更多工具调用
  │
  ▼
interpreter_generator
  │
  ├── 读取工具结果、HVAC 上下文、多意图计划
  └── 生成最终 Markdown 报告
  │
  ▼
结束
```

这一流程体现了典型 ReAct 模式：**思考 → 调工具 → 观察结果 → 再思考 → 生成答案**。

## 7. 模块设计

### 7.1 Graph 调度层

相关文件：

- `src/graph/state.py`
- `src/graph/builder.py`
- `src/graph/nodes.py`
- `src/graph/edges.py`

Graph 层定义了 Agent 的状态结构和节点流转逻辑。

`AgentState` 主要承载：

- `user_input`：用户原始输入。
- `messages`：LangChain 消息列表。
- `hvac_knowledge`：HVAC RAG 检索结果。
- `constraints`、`physics_verification`：为能源调度和物理校验预留的数据字段。
- `page_context`：前端传入的当前页面、站点等上下文。
- `pending_actions`：待前端执行的动作列表。
- `intent_plan`：多意图执行计划。
- `final_report`：最终解释报告。
- `error`：错误信息。

三个核心节点分别是：

| 节点 | 职责 |
| --- | --- |
| `cognitive_parser_node` | 加载系统 Prompt、注入路由表和页面上下文，调用 LLM 判断工具调用 |
| `v3_engine_router_node` | 执行工具、写入状态、匹配并调度 Skill |
| `interpreter_generator_node` | 根据工具结果和上下文生成 Markdown 报告 |

### 7.2 Skills 业务推理层

相关文件：

- `src/skills/base_skill.py`
- `src/skills/__init__.py`
- `src/skills/hvac_expert_skill.py`
- `src/skills/ui_router_skill.py`
- `src/skills/energy_dispatch_skill.py`
- `src/skills/v3_interpreter_skill.py`

`BaseSkill` 抽象基类规定了所有 Skill 的统一接口：

- `name`：技能名称。
- `tools`：该技能关注的工具列表。
- `prompt_keys`：该技能依赖的 Prompt 配置。
- `description`：技能描述。
- `execute()`：核心编排逻辑。
- `before_execute()` / `after_execute()`：生命周期钩子。

当前注册的四个 Skill：

| Skill | 状态 | 职责 |
| --- | --- | --- |
| `HVACExpertSkill` | 已实现 | 根据 RAG 结果决定正常回答或低置信拒答，并注入引用格式 |
| `UIRouterSkill` | 已实现 | 根据工具调用结果生成 `UIAction` 页面跳转动作 |
| `EnergyDispatchSkill` | 骨架 | 为负荷预测、光伏协同、储能调度等 PowerAI 能力预留 |
| `V3InterpreterSkill` | 骨架 | 为后续将报告生成逻辑从 Graph 节点迁移到 Skill 预留 |

这种设计把“工具执行”和“业务判断”分开，降低了主图节点复杂度，也方便后续新增垂直领域 Skill。

## 8. HVAC RAG 技术方案

HVAC 问答是当前项目中最完整的 AI 知识能力之一。

### 8.1 数据入库

相关文件：

- `src/pipelines/rag_ingest.py`
- `src/tools/query_hvac_knowledge.py`

RAG 入库流程：

1. 读取 HVAC 问答语料。
2. 清洗文本内容。
3. 识别系统类型。
4. 使用 `BAAI/bge-small-zh-v1.5` 生成中文向量。
5. 写入本地 ChromaDB 持久化目录。

项目文档中记录当前知识库约 5605 条 HVAC 语料。

### 8.2 检索与质量控制

`query_hvac_knowledge` 执行时会：

1. 使用相同 Embedding 模型对用户问题向量化。
2. 从 ChromaDB 中检索候选文档。
3. 多取若干条结果，用于后续去重。
4. 根据 embedding 余弦相似度执行 MMR 风格去重。
5. 按 `top_k` 截断结果。
6. 根据 `confidence_threshold` 判断是否低置信。
7. 构造 `source_snippets`，供最终回答标注来源。

### 8.3 低置信拒答

如果检索结果距离超过阈值，`HVACExpertSkill` 会触发低置信拒答：

- 向报告生成节点注入拒答 Prompt。
- 替换检索上下文，避免 LLM 看到无关片段后强行编造。
- 引导模型说明“当前知识库未检索到足够可靠依据”。

这解决了 RAG 系统常见的“无依据仍然回答”问题。

### 9.2 Token 自动刷新

相关文件：

- `src/utils/fuca_token_refresher.py`
- `src/tools/java_backend.py`

Token 处理逻辑：

1. 工具调用时从环境变量读取福加 API Token。
2. 调用真实后端接口。
3. 如果返回 401，尝试通过登录流程刷新 Token。
4. 登录密码使用 RSA 公钥加密。
5. 刷新成功后更新 `.env` 中的 Token。
6. 使用新 Token 自动重试原请求。

该设计减少了演示和联调时因 Token 过期导致的中断。

## 10. 页面导航与 UIAction 设计

页面跳转能力由三部分组成：

- `config/routes.yaml`：维护真实前端路由。
- `src/tools/navigate_to_page.py`：构造跳转动作。
- `src/skills/ui_router_skill.py`：根据工具调用结果推断是否跳转。

当前路由配置区分：

- **可访问路由**：约 24 个，可直接下发跳转。
- **受限路由**：约 10 个，用于识别但不直接跳转，并说明限制原因。

`UIRouterSkill` 生成跳转动作有两个优先级：

1. 如果 LLM 显式调用 `navigate_to_page`，优先采用该结果。
2. 如果 LLM 调用的是业务数据工具，则根据工具到页面的映射自动推断路由。

例如用户问“查看当前 COP”，Agent 可以同时：

- 调用 `fetch_cop_data` 查询数据。
- 生成文字解释。
- 下发跳转到对应监控页面的 `UIAction`。

## 11. 多意图识别机制

项目 Phase 7 已实现多意图识别。

当 `cognitive_parser_node` 发现 LLM 一次返回多个 `tool_calls` 时，会自动构造 `IntentItem` 列表：

- `id`：意图编号。
- `description`：例如“调用 fetch_cop_data”。
- `category`：根据工具映射为 `hvac`、`monitor`、`alarm`、`export` 等类别。
- `status`：当前状态，初始为 `pending`。

后续报告生成时，`interpreter_generator_node` 会把多意图计划注入 Prompt，引导模型按意图分段输出，避免多个任务混成一段不清晰的回答。

SSE 接口也会推送 `intent_plan` 事件，方便前端展示执行计划。

## 12. SSE 前后端交互协议

`POST /stream` 是正式前端集成的关键接口，返回 `text/event-stream`。

当前支持事件：

| 事件 | 含义 | 前端建议 |
| --- | --- | --- |
| `thinking` | 认知解析阶段的思考文本 | 可折叠展示或忽略 |
| `tool_call` | 即将调用的工具名和参数 | 可用于展示执行过程 |
| `tool_result` | 工具返回结果 | 可折叠展示或用于调试 |
| `rag_sources` | HVAC 知识库检索来源 | 可展示引用来源 |
| `text` | 最终回答文本流 | 主体回答区域增量渲染 |
| `intent_plan` | 多意图执行计划 | 可展示任务拆解列表 |
| `action` | 前端动作，例如页面跳转 | 调用前端路由执行 |
| `error` | 错误信息 | 展示错误提示 |
| `done` | 流结束 | 关闭流式状态 |

典型请求体：

```json
  "user_input": "查看当前站点 COP",
  "page_context": {
    "current_route": "/integrated-monitor/energy-monitor",
    "site_id": "default"
  }
}
```

```text
event: action

```

## 13. MCP 算法模型层设计

`MCP_INTERFACE_SPEC.md` 定义了 EnerGraph 与算法模型层的接口契约。该能力当前属于规划和接口设计阶段，真实算法服务尚未完成接入。

规划中的算法模型能力包括：

- 负荷预测。
- 光伏预测。
- 储能调度优化。
- 物理约束校验。
- 碳排与收益评估。

设计原则是：**决策层不直接实现数值优化算法，而是通过 MCP Client 调用算法模型层服务。**

这样做的好处：

- 算法模型可独立开发、部署和升级。
- Agent 核心流程不需要频繁修改。
- 新增算法能力时，只需注册新的工具或 MCP Server。
- 有利于未来接入更多能源、碳管理、设备控制模型。

## 14. 安全、约束与边界

当前项目在安全和边界上体现了几个原则：

1. **读多写少**：当前主要能力是查询、解释和导航，不直接下发设备控制指令。
2. **页面受限路由识别**：`routes.yaml` 中区分可访问和受限路由，避免任意跳转。
3. **API 可选鉴权**：FastAPI 支持 Bearer Token，开发环境可关闭。
4. **Token 不直接写入代码**：通过环境变量和 `.env` 管理。
5. **RAG 低置信拒答**：避免知识库无依据时编造答案。
6. **高风险控制预留人工确认**：规划文档中建议设备控制类操作引入 Human-in-the-Loop，但当前尚未落地为主流程。

项目当前边界也需要明确：

- EnerGraph 当前不直接控制设备启停。
- PowerAI 储能调度尚未完成真实算法接入。
- MCP 算法服务目前是契约和规划，不是已联通的生产能力。
- Agent 子图基础设施已经存在，但主图当前仍主要通过 Skill 调度工具结果。
- Phase 5 语音助手、Phase 6 可视化报表仍处于计划阶段。

## 18. 对外介绍话术

### 18.1 30 秒版本

EnerGraph 是福加能碳管理平台的能源管理 AI Agent，基于 LangGraph 构建，定位在青山大模型 V3.0 的决策层。它可以理解用户自然语言问题，自动选择 HVAC 知识库、福加后端 API 或页面跳转工具，查询真实运营数据并生成解释报告。当前已经支持 HVAC RAG 问答、COP、能耗、报警、碳排、光伏等数据查询，以及 FastAPI SSE 前端对接和页面导航信号下发。

### 18.2 3 分钟版本

EnerGraph 的核心定位不是一个普通聊天助手，而是能源管理系统里的智能编排大脑。用户可以用自然语言提出业务问题，例如查看 COP、查询报警、分析能耗、了解 HVAC 节能建议，系统会先通过 `cognitive_parser` 判断用户意图，再通过 LangGraph ReAct 循环选择合适工具。

在架构上，它分为 Graph、Skills、Tools、Schemas、Config 和 API 几层。Graph 负责流程调度，Skills 负责业务 SOP，Tools 负责确定性执行，Schemas 保证数据契约，Prompt 和路由配置则外置管理。当前主图包含认知解析、工具调度和报告生成三个节点，工具执行后会统一进入 BaseSkill 体系，例如 HVAC Skill 会判断 RAG 结果是否低置信，UI Router Skill 会根据查询结果生成前端页面跳转动作。

在技术实现上，项目已经完成 HVAC 知识库 RAG，使用 ChromaDB 和中文 Embedding 模型，支持相似度阈值、去重、低置信拒答和引用来源。它也已经封装福加 Java 后端多类真实接口，包括 COP、能耗、报警、碳排、光伏、设备排名、环境参数和能效数据。前端集成方面，项目提供 FastAPI `/invoke` 和 `/stream` 接口，SSE 能推送思考过程、工具调用、工具结果、RAG 来源、最终文本、多意图计划和页面动作。

未来 EnerGraph 会继续向 PowerAI 能源调度演进，通过 MCP 接入负荷预测、光伏预测、储能调度优化和物理校验模型。同时会把报告生成进一步 Skill 化，引入 Agent 子图、可视化报表、人机确认和更强的 RAG 检索能力。

## 19. 汇报时可强调的核心价值

1. **降低平台使用门槛**：用户不用记菜单和指标口径，直接用自然语言查询。
2. **连接真实业务系统**：已经对接福加真实 API，而不是停留在静态 Demo。
3. **兼顾解释与动作**：既能生成报告，也能驱动前端页面跳转。
4. **架构可扩展**：Skill、Tool、Prompt、Schema 分层，便于新增业务能力。
5. **面向算法模型层演进**：通过 MCP 预留预测、优化和物理校验能力。
6. **交互透明**：SSE 可展示工具调用、检索来源和执行计划，便于用户信任 Agent。
